## 这篇文章要回答什么？

- 同一份原型可以跨宿主运行，这件事到底意味着什么。
- 这种能力会如何改变 Proto UI 的消费方式。
- 它会给原型层和适配器层分别带来什么职责变化。

## 跨宿主运行不是附属特性

- 它不是“顺便支持多几个框架”的附加能力。
- 它是 Proto UI 最核心的消费方式之一。
- 对很多使用者来说，他们真正接触 Proto UI 的方式，就是在不同宿主中消费同一份原型。

## 这件事的价值不只是“到处都能跑”

- 更重要的是，分离之后不同层可以朝不同方向演进。
- 原型层可以更专注交互表达与语义设计。
- 适配器层可以更专注宿主解释力、整合能力和性能优化。
- 生态不必再把“组件定义”和“组件实现”永远绑在一起。

## 这会影响哪些人？

- 对组件使用者：可以用统一的原型能力面向不同宿主消费组件。
- 对原型作者：关注点更偏向交互语义，而不是某个框架实现技巧。
- 对适配器作者：关注点更偏向如何把语义高质量地解释到宿主环境中。

## 这件事也有边界

- 它不意味着所有宿主天然完全等价。
- 它不意味着可以零成本迁移一切组件。
- 它不意味着只要有原型就会自动得到完美一致的落地结果。

跨宿主运行是 Proto UI 的能力方向，但它依然需要清晰的能力边界、解释层设计和工程约束。
