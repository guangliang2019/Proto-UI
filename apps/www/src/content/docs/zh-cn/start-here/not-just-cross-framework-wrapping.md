## 这篇文章要回答什么？

- Proto UI 和常见的跨框架封装、包装层、代码生成到底有什么本质区别。
- 为什么 Proto UI 关心的重点不是“少写几套代码”。
- 为什么这种差异会影响整个组件生态的分工方式。

## 读者通常会先有的疑问

- 这是不是一种更复杂的 adapter 模式？
- 这是不是把同一套组件包装成多个框架版本？
- 这是不是一套更自动化的 codegen 流程？

这些疑问都很自然。  
这篇的任务不是否定这些联想，而是划清 Proto UI 的目标边界。

## 普通的跨框架封装通常在做什么？

- 以某个具体实现为中心，向外适配多个框架。
- 目标通常是复用实现、减少重复开发、统一 API 表面。
- 即使支持多个框架，组件的语义主体仍然依附于具体实现。

## Proto UI 试图分离的是什么？

- 不是先有某个框架组件，再把它翻译到别处。
- 而是先定义一份独立的交互原型，再由不同宿主去解释它。
- 也就是说，被分离出来的是组件的**交互语义层**。

## 这件事为什么重要？

- 复用的不再只是代码，而是交互定义本身。
- 原型层可以更专注于“组件应该如何交互”。
- 适配器层可以更专注于“如何在特定宿主中更好地解释、优化和落地这些语义”。
- 一旦这层分离成立，生态组织方式就会发生变化。
