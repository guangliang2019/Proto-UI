## 这篇文章要回答什么？

- 为什么组件生态会需要 Proto UI 这样一层协议化尝试。
- 它试图解决的到底是哪一类问题。
- 为什么这个问题不是某个框架单独能解决的。

## 当前的问题背景

- 前端生态并不缺框架，也不缺组件库。
- 真正困难的是：当框架、设计语言和平台交叉时，组件实现会成倍增长。
- 我们今天主要依赖复制、移植和重复实现来对抗这种复杂度。

## Proto UI 认为问题出在哪里？

- 问题不只是“组件太多”，而是“交互语义被长期绑定在具体实现中”。
- 组件看起来属于某个框架，但其中很多核心语义并不天然属于那个框架。
- 一旦交互语义无法独立表达，复用就只能围绕实现层展开。

## Proto UI 的尝试是什么？

- 不再把组件完全视为某个框架下的实现产物。
- 而是尝试先定义独立的交互原型，再由不同宿主去解释和落地。
- 它要回答的是：组件是否可以先作为协议存在，再作为实现被消费。
