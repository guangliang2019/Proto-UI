## 这篇文章要回答什么？

- Proto UI 的原型为什么不是静态结构，而是分时期成立的交互体。
- 为什么 Proto UI 必须区分 `setup` 与 `runtime`。
- 为什么生命周期不是附属能力，而是交互语义成立的关键部分。

## 原型不是静态描述

- Proto UI 中的原型，不只是“组件长什么样”的结构描述。
- 它描述的是一个会在时间中展开、会在不同阶段承担不同职责的交互主体。
- 因此，执行语义是原型成立的必要条件，而不是补充细节。

## 为什么要区分 `setup` 与 `runtime`？

- `setup` 是定义、计划、声明交互行为的时期。
- `runtime` 是组件在宿主中真正运行、状态变化、上下文流动与副作用发生的时期。
- 这两个时期的职责完全不同，因此 Proto UI 必须对它们作出清晰分割。

## 这种分期为什么重要？

- 它避免“描述行为”和“执行行为”混杂在一起。
- 它帮助原型保持可理解、可翻译、可验证的结构。
- 它使适配器能够在明确的时期边界内解释原型，而不是在模糊时机中猜测作者意图。

## 生命周期在这里扮演什么角色？

- 生命周期不是某个 GUI 框架的借用概念，而是组件作为交互体天然具有的时间属性。
- 一旦组件要在时间中存在，它就必然会拥有阶段推进、能力开启与能力释放的问题。
- Proto UI 用生命周期来组织这些时间维度上的交互语义。

## 这里适合在正式草稿中展开什么？

- `setup` 期间更偏向声明与计划
- `runtime` 期间更偏向实际读取、变化与副作用
- 为什么某些能力天然属于某一时期，而不属于另一时期

## 这篇不打算展开什么？

- 不在这里列出完整的时期 API 表。
- 不在这里细化适配器生命周期门控点。
- 不在这里替代工程章节中的运行时架构说明。

## 这篇要把读者带到哪里？

- 让读者理解 Proto UI 的执行语义是一种结构性约束。
- 让读者理解生命周期与执行时期，是交互语义的一部分。
- 带着这层理解继续进入 `翻译层：Adapter / Compiler`。
