## 这篇文章要回答什么？

- Proto UI 的原型为什么不能随意拆分或合并。
- 一个子结构什么时候必须被视为独立原型。
- 为什么“是否激活信息通路”会成为原型拆分的判断依据。

## Proto UI 为什么关心原型拆分？

- 原型不是为了描述视觉树，而是为了描述交互主体。
- 如果拆分缺乏原则，原型就会退化成任意切块的结构描述。
- Proto UI 需要一套稳定标准，判断一个子结构是否已经形成独立的交互责任边界。

## 原型拆分的核心判断是什么？

- 当一个子结构存在除 `feedback` 之外的任一信息通路需求时，它必须拆成独立原型。
- 当一个子结构仅激活 `feedback` 通路时，它可拆可不拆。
- 当一个子结构没有激活任何信息通路时，它必须不拆。

## 为什么 `feedback-only` 可以不拆？

- 因为 `feedback` 更接近呈现与感知层。
- 它本身并不天然意味着这个子结构已经形成独立交互主体。
- 只有当一个子结构开始承担输入、配置、暴露、上下文协作等责任时，它才真正拥有独立的原型意义。

## 这里适合在正式草稿中展开哪些例子？

- 子结构响应 `event`
- 子结构拥有独立 `props`
- 子结构向外暴露自身状态或方法
- 子结构订阅或分发 `context`

这些例子都能说明：  
一旦子结构开始承担独立的信息通路责任，它就不再只是父原型的一段内部结构。

## 这篇不打算展开什么？

- 不在这里定义每条通路的正式语义。
- 不在这里写原型 API 的具体写法。
- 不在这里讨论适配器如何消费拆分后的原型。

## 这篇要把读者带到哪里？

- 让读者理解 Proto UI 的原型粒度是有原则约束的。
- 让读者理解“独立原型”意味着独立交互责任边界。
- 带着这层理解进入 `执行语义`。
