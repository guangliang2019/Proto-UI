## 这篇文章要回答什么？

- 为什么 Proto UI 的原型不能直接等同于宿主实现。
- 为什么原型需要通过翻译层才能落地。
- 为什么翻译天然可能有损，而这种有损并不否定 Proto UI 的哲学。

## 为什么需要翻译层？

- 原型表达的是交互语义，不是某个宿主下的最终实现。
- 不同宿主拥有不同能力边界、性能约束与最佳实践。
- Proto UI 需要一个中间层，把原型解释为宿主可接受的结果。

## Adapter 与 Compiler 在这里分别意味着什么？

- **Adapter**：在运行时或宿主侧解释原型。
- **Compiler**：在更静态的阶段把原型转换为目标宿主的产物。

这里适合在正式草稿中展开：

- 为什么当前阶段以 Adapter 为主路径
- 为什么 Compiler 是 Proto UI 的后续主要方向
- 两者如何共享同一套语义基线

## 翻译为什么天然可能有损？

- 因为宿主能力边界并不相同。
- 因为不同宿主对交互、渲染、引用与副作用管理的方式并不相同。
- Proto UI 不承诺抹平所有宿主差异，只承诺尽可能维持语义上的对应关系。

## 宿主边界意味着什么？

- 原型的可还原程度，最终受制于宿主能力极限与现实工程难处。
- 同一原型在不同宿主上，允许存在表现层面的差异。
- 但这种差异不应任意破坏 Proto UI 所定义的核心交互语义。

## 这里适合在正式草稿中强调什么？

- 同属 Web 技术体系的宿主，原则上应追求更高的 `feedback` 一致性。
- 跨平台宿主之间允许存在更明显的反馈差异。
- `host` 通路默认不属于 Proto UI 的跨平台主承诺范围。

## 这篇不打算展开什么？

- 不在这里写 Adapter API 指南。
- 不在这里列出宿主能力矩阵。
- 不在这里替代工程章节中的 `Host Caps` 与 `Adapter Guide`。

## 这篇要把读者带到哪里？

- 让读者理解为什么翻译层既必要又不可避免地面对有损问题。
- 让读者理解宿主边界不会否定 Proto UI，反而说明翻译层的重要性。
- 带着这层理解继续进入 `设计约束`。
