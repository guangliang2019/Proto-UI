## 这篇文章要回答什么？

- 如果要描述一个组件的全部交互，最自然的切分方式是什么。
- Proto UI 为什么以“使用者”和“信息通路”来组织原型语法。
- 为什么 `feedback`、`event`、`props`、`expose`、`context` 会成为原型的核心子能力。

## Proto UI 如何切分组件的使用者？

Proto UI 认为组件至少面对三类典型使用者：

- **User**
- **Maker**
- **Other Component**

这三个角色并不是产品角色，而是哲学意义上的交互抽象。

## 什么是信息通路？

- 组件与使用者之间进行信息传递的路径，被 Proto UI 称为“信息通路”。
- 原型的很多核心能力，本质上都来自对这些通路的抽象。
- 信息通路不是 API 清单，而是 Proto UI 组织原型语法的主要维度。

## 由信息通路可以导出哪些核心能力？

- `feedback`
- `event`
- `props`
- `expose`
- `context`

这里适合在正式草稿中展开：

- 每一条通路分别对应哪一类交互关系
- 为什么这些子能力不是随意枚举出来的
- 为什么它们在 Proto UI 中属于“必须存在”的能力

## 信息通路之外还有什么？

- 信息通路并不足以完全描述组件的全部。
- 组件仍然拥有一些固有属性，例如时间属性、空间属性与自描述信息。
- 因此 Proto UI 还需要 `state`、`lifecycle`、`meta` 这样的能力维度。

## 信息通路可以扩展吗？

- 可以。
- 当组件新增稳定且必要的使用者类别时，新的信息通路就可能成立。
- 例如与宿主直接交互的 `host` 通路，就是可能存在但默认不作为跨平台主轴的能力方向。

## 这篇不打算展开什么？

- 不在这里定义各通路的正式契约。
- 不在这里讨论每个子能力的工程 API。
- 不在这里讨论原型应当如何拆分。

## 这篇要把读者带到哪里？

- 让读者理解 Proto UI 为什么能从“使用者关系”推导出原型能力。
- 让读者理解信息通路模型不是分类技巧，而是 Proto UI 的组织原理。
- 带着这套模型继续进入 `原型拆分原则`。
