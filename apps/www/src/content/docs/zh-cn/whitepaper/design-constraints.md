## 这篇文章要回答什么？

- Proto UI 为什么不是一个无限开放的抽象工程。
- 它主动接受了哪些约束。
- 这些约束如何保护 Proto UI 的长期可移植性与可验证性。

## 为什么需要设计约束？

- 如果一个协议化系统什么都想表达，它很快就会失去边界。
- Proto UI 不是为了把所有 GUI 问题吞并进核心层，而是为了保住最关键的交互语义。
- 因此，它必须用设计约束保护自己的抽象质量。

## Proto UI 现阶段的约束倾向是什么？

- 倾向可序列化的数据表达
- 倾向把宿主特有能力隔离在边界层
- 倾向把跨平台主轴建立在稳定、可迁移的信息通路之上

## 为什么有些能力不进入核心哲学主轴？

- 因为它们天然高度依赖宿主
- 因为它们难以跨平台迁移
- 因为如果过早纳入核心层，会削弱 Proto UI 的一致性目标

这里适合在正式草稿中展开：

- `host` 通路为什么重要但不进入核心白皮书主轴
- 为什么函数、平台对象等能力不适合作为协议层默认语义
- 为什么 Proto UI 更愿意先保守，再逐步放宽边界

## 这些约束不是保守主义

- 它们不是因为 Proto UI 能力不足而做出的临时退让。
- 它们是为了让协议层能够长期成立、被验证并被迁移。
- 真正的克制，是 Proto UI 与“什么都想统一”的工程冲动之间的分界线。
