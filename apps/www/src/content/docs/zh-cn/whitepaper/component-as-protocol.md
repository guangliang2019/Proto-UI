## 这篇文章要回答什么？

- Proto UI 所理解的“组件”究竟是什么。
- 为什么组件可以脱离具体技术实现，被当作一种可描述的交互体来看待。
- 为什么 Proto UI 会把这种可描述对象称为“原型”，并把其落地机制称为“适配器”。

## Proto UI 认为什么是组件？

- 组件首先不是某个框架里的 class、function 或 object。
- 组件首先是一种能够与使用者进行信息交流的“交互体”。
- 它可以是物理的、抽象的、图形化的，甚至在不同宿主里拥有不同表现形式。

Proto UI 更关心的是：  
一个对象是否“可交互”，而不是它以哪种技术形态存在。

## 为什么组件可以被协议化理解？

- 如果一个组件的交互语义能够被充分描述，那么它就不必依附于某个单一实现而存在。
- 真正决定组件可还原程度的，不是 Proto UI 是否相信“描述”，而是宿主本身的能力边界。
- 即使同一组件在不同宿主上允许有不同交互表现，它依旧可以拥有一个更高层的语义主体。

## Proto UI 如何命名这些层次？

- **Prototype（原型）**：组件交互本质与固有属性的描述。
- **Adapter（适配器）**：将原型翻译到指定宿主的解释层。
- **Host（宿主）**：承载 Proto UI 落地结果的技术或物理环境。

这里适合在正式草稿中展开：

- 为什么 Proto UI 使用“宿主”这一命名
- 为什么它带有一种“寄生于宿主实现”的意味
- 为什么这并不意味着 Proto UI 依附性弱，反而说明它是更上层的思想

- 让读者先接受“组件可作为协议被理解”这一第一性原理。
