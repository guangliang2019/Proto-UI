## 摘要

**Proto UI v0 是协议语义闭环阶段。**  
核心信息通路与执行语义趋于稳定，并已在多个宿主环境中完成验证。  
当前以 Adapter 为主要落地方式，Compiler 将在 v1 成为主路径。

## 阶段对比

| 维度 | v0                             | v1                                |
| ---- | ------------------------------ | --------------------------------- |
| 本质 | 协议语义闭环                   | 稳定性与兼容承诺                  |
| 能力 | 原型可翻译为多个 Web 框架组件  | 原型可翻译或编译为 Web 与原生目标 |
| 生态 | 官方无样式组件库与参考 Adapter | 系统化沉淀主流组件语义            |

> 表格用于概括阶段差异，正式定义以本文文字说明为准。

## 当前阶段

Proto UI 目前处于 v0 阶段。

v0 以“协议语义闭环”为目标：  
核心语义趋于稳定，协议模型可被验证，并具备跨宿主落地能力。

这不是一个功能完备的 UI 框架版本。  
它是协议模型从概念走向可验证实现的阶段性节点。

## v0 的目标

v0 的核心目标是验证以下几点：

- 信息通路模型在结构上完整。
- 执行语义保持稳定。
- Prototype 可以在多个宿主环境中落地，而不发生语义漂移。
- 契约驱动架构在实践中可行。

v0 的意义在于确认协议模型成立。

---

## v0 的保证范围

以下内容在 v0 阶段视为稳定：

### 1. 执行语义

- `exec-phase` 的语义被视为稳定。
- 与生命周期相关的能力守卫不会发生破坏性变更。

### 2. 核心信息通路

- `props`、`event`、`expose`、`feedback`、`state`、`context` 的核心语义冻结。
- 既有契约测试所定义的行为预期不会被重新解释。

允许新增子 API，但不会改变现有语义含义。

### 3. 跨宿主验证

- 至少两个 Adapter 提供完整的 Prototype 执行能力。
- 核心模块具备契约测试覆盖。

### 4. 官方基础原型

- 官方 base prototype 可覆盖常见 headless UI 组件能力。
- 其行为可作为 Adapter 的语义参考基线。

## v0 的设计原则

### 可序列化的数据约束

所有接受或存储数据的信息通路，默认应承载可序列化的数据。

- 函数与平台对象不属于协议层语义的一部分。
- Adapter 可在边界层提供宿主环境友好的形式，但核心语义保持可序列化。

该约束为跨平台能力与未来 Compiler 形态预留空间。

### Adapter 的角色

Adapter 被视为协议语义向宿主环境的“有损翻译层”。

Adapter 可以优化开发体验，但不得重新定义协议语义。

## v0 不承诺的内容

- 工业级别的向后兼容保证。
- 完整的性能优化。
- 成熟的 Adapter 生态体系。
- 与未来 Compiler 的完全一致性保证。

v0 阶段可能提供向后兼容补丁，但正式的稳定性承诺从 v1 开始。

## 与 v1 的关系

v0 通过 Adapter 验证协议模型。

v1 将引入 Compiler 作为主要执行路径。

在 v1 阶段：

- 原型语法提供稳定性与向后兼容承诺。
- Compiler 成为推荐的生产形态。
- 官方 Adapter 与 Compiler 输出保持语义一致。
- 性能优化主要通过编译期实现。

---

## 总结

v0 确认协议语义成立并可迁移。  
v1 在此基础上提供稳定性承诺与生成能力。

Proto UI 不旨在取代现有组件库，而是为其提供协议级表达与跨平台能力。
