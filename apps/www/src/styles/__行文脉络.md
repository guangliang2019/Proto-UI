# Proto UI · Overview 行文大纲

> 目标：建立认知路径，而非讲解全部细节。顺序：问题 → 定义 → 阶段 → 结构 → 导航

---

# 1. Why Proto UI

## 目标

提出问题，引发读者思考“组件是否必须依附框架”。

## 行文结构

1. 行业现象（跨框架重复实现组件）
2. 更深层问题（语义绑定在实现中）
3. 核心提问（组件是否只能属于某个框架？）
4. 假设（如果语义可迁移，是否可以协议化？）
5. 冷静定位（一句话说明 Proto UI 的方向）

## 必须回答

- 重写的不只是样式，而是语义
- 迁移成本来自实现绑定
- 协议是一个“可能的解法”

## 禁止展开

- exec-phase
- module 结构
- contract 细节
- 内部术语定义

# 2. What is Proto UI

## 目标

给出极简定义，不引入复杂结构。

## 行文结构

1. 一句话定义（Proto UI 是什么）
2. Prototype 的角色（抽象语义）
3. Adapter 的角色（翻译到宿主）
4. Compiler 的方向（未来生成路径）
5. 当前阶段一句话说明

## 必须回答

- 它不是框架
- 它不是代码生成工具
- 它是协议化表达

## 禁止展开

- 生命周期细节
- 模块划分
- 内部执行机制

# 全局写作约束

1. Overview 中不定义所有术语。
2. 不使用内部工程词汇。
3. 每篇尽量控制在 800–1000 字以内。
4. 每篇必须有一条“可以被引用的核心句”。
5. 不用抽象修辞掩盖技术事实。
6. 不写愿景宣言。
7. 不和任何框架形成对抗叙事。
