# Value 类组件的实践探索路线

## 这份文档的目的

本文不是为了立即定义一套新的正式 API，也不是为了提前抽象出通用能力。

当前目标更具体：

- 明确 Proto UI 在 `value` 类组件上的探索重点；
- 区分哪些场景真正需要 `run.update`；
- 为原型库扩充阶段提供一条较稳的实践路线；
- 记录若干阶段性判断，避免后续讨论反复回到同一前提混乱之中。

当前阶段的核心任务不是“做出更多组件”，而是通过一批有代表性的原型，摸清 Proto UI 在**值变化 -> UI 回显**这一链路上的能力边界。

---

## 问题背景

在现有原型库中，绝大多数组件并未真正使用 `run.update`。这带来两个直接现象：

1. 贡献者可能误以为 Proto UI **不支持 update**；
2. 原型实现会自然滑向另一种思路：  
   当组件需要展示当前值时，不在 Proto UI 内部重跑模板，而是直接将值 expose 给 App Maker，由宿主层自行消费与渲染。

这种思路并非不可行，甚至在工程上相当干净：

- Proto UI 本身更轻；
- 不必过早承担模板重算与调度复杂度；
- 更容易嵌入宿主现有渲染体系。

但它也带来一个关键代价：

> 组件中一部分本应由 Prototype Author 定义的交互与呈现，被泄露到了 App Maker 与宿主层。

这与 Proto UI 希望“对组件的完整交互进行定义”的方向并不完全一致。

因此，当前需要验证的不是“要不要把所有 value 都交给 update”，而是：

> 哪些值变化必须由 Prototype 内部定义并驱动 UI 回显，哪些则可以安全地留给宿主扩展。

---

## 当前阶段的几个工作假设

以下内容应视为**工作假设**，而非已经冻结的正式结论。

### 1. `run.update` 必须存在，而且不能只是名义能力

Proto UI 不能长期停留在“理论上支持 update，但原型库几乎不用”的状态。

原因不是为了追求形式完整，而是因为某些非常自然的组件场景，确实无法仅靠 expose + 样式系统闭合，例如：

- 当前值文本；
- 选中值摘要；
- 计数与数量标签；
- 页码、百分比、状态文案；
- anatomy 中的 `*-value` 相关区域。

这些都属于组件完整交互的一部分，而不是宿主层的随机增强。

### 2. `run.update` 不应演化为 Proto UI 自己的宿主级调度系统

Proto UI 需要拥有“语义级重算权”，但不应试图复制宿主框架的调度机制。

更合理的边界是：

- Proto UI 定义：
  - 何时语义上需要重新计算模板；
  - 模板消费哪些动态值；
  - 重算后输出哪些宿主中立的结构与语义；
- Adapter / Host 负责：
  - 用宿主友好的方式提交这次重算；
  - 尽量嵌入宿主已有的更新与调度路径。

也就是说，Proto UI 应保留**定义权**，但不必强行掌握**宿主执行权**。

### 3. “是否需要 update”应按值的消费方式来区分，而不是按组件名区分

这比“先做哪些组件”更重要。

目前可以先把场景分成两类：

#### A. Exposed-state 型

值变化后，模板本身不一定需要重算，主要由 adapter 将内部状态暴露给样式系统或其他宿主机制消费。

例如：

- Slider
- Rating
- 某些 Toggle / Switch / Pressed 状态类组件

这类原型有价值，但其主要探索目标不是 `run.update`，而是：

- 内部 state 暴露是否足够；
- adapter 如何将 exposed state 交给样式系统；
- 交互状态与样式反馈是否一致。

#### B. Render-consumed-value 型

值变化后，Proto UI 自身定义的模板明确消费了这个值，因此必须重跑 render 或执行某种正式的模板重算。

例如：

- Stepper
- Number Input 中的 value 区域
- 各类 `*-value`
- 选中值摘要
- 计数/数量/页码文本

这类原型才是当前阶段探索 `run.update` 的核心样本。

---

## 当前探索重点：优先选择 Render-consumed-value 型原型

### 为什么不是先做 Slider / Rating

Slider 与 Rating 的交互很有代表性，但它们不必天然成为 `run.update` 的优先探针。

原因很简单：

- 它们的值往往可以只存在于内部 state 中；
- render 本身不一定直接消费该值；
- UI 变化可能主要通过 expose + 样式系统完成；
- 即使值暴露给外部，也不一定要求 Proto UI 内部模板重算。

所以，Slider / Rating 更适合作为**次级验证样本**，用于观察：

- state expose 是否足够；
- adapter 是否能稳定消费这些状态；
- Proto UI 是否需要进一步规范交互态与样式态的桥接。

它们不是当前阶段验证 `run.update` 的首要对象。

---

## 第一批建议实践的原型

### 1. Stepper

优先级最高。

原因：

- 它是最小的 `render-consumed-value` 闭环；
- 既有内部状态变化，又有明确的文本/属性回显；
- 交互相对可控，不容易被宿主原生输入行为干扰；
- 能最快暴露 `run.update` 的真实缺口。

重点观察：

- 值变化后由谁触发 update；
- render 重跑是否能稳定更新 value 文本；
- 连续点击时更新顺序是否清晰；
- `disabled`、`min`、`max` 等相关语义是否能同步回显；
- 同一次交互驱动多个模板位置变化时，是否存在时序裂缝。

### 2. `*-value` 类呈现部件

这不必一开始就绑定到完整 Slider、Input、Select 上。

更适合把它单独当作一类“值显示部件”来验证，例如：

- `StepperValue`
- `ProgressValueText`
- `SelectedValueSummary`
- `CountLabel`

重点不是组件名，而是验证：

- 当前值是否属于 prototype 的正式定义范围；
- value 区域是否需要自定义渲染逻辑；
- 这类区域是否总是要求模板重算；
- 它们是否会逐渐形成某种稳定 anatomy 模式。

### 3. 带选中值摘要的离散选择器

例如：

- Segmented Control + 当前选中值文本
- Radio Group + “当前选择：X”
- Tabs + 当前选中标签名 / 索引回显

这类原型的价值在于：

- 交互不复杂；
- 值变化会驱动多个模板位置联动；
- 既不像输入控件那样深度依赖宿主原生行为，也比 Stepper 更接近真实组件使用场景。

### 4. Count / Badge / 当前页文本等外围值展示

例如：

- `3 selected`
- `Page 2 / 10`
- 标题后的计数标签
- 当前步骤编号

这类原型通常不是“控件本体”的中心，但非常适合暴露一个关键问题：

> Proto UI 是否只擅长定义核心控件行为，而对外围 value presentation 支持不足？

---

## 当前阶段不建议过早触碰的原型

### 1. Input / TextArea

暂缓的原因不是它们不重要，而是它们会过早引入大量与当前目标无关的复杂度：

- 原生输入行为强耦合；
- composition / IME / selection 等问题复杂；
- 各宿主的输入模型不对称；
- 按 Proto UI 的 philosophy 去拆 anatomy，原生 input 往往天然耦合多个交互单元；
- Native 与 non-native 两种实现路径的决策仍需后续验证。

在 Proto UI 自己的 update 机制、值呈现机制尚未摸清前，不宜过早投入。

### 2. 复杂宿主依赖型 value 组件

例如：

- 原生 Select / 非原生 Select 的完整决策；
- DatePicker / TimePicker；
- 高耦合文本输入体系。

这些组件会把“update 机制”问题与“宿主输入模型”问题缠在一起，干扰判断。

---

## 关于 `run.update` 的阶段性理解

当前更合理的理解不是：

> `run.update` 会让 Proto UI 自己掌握一套完整渲染系统。

而是：

> `run.update` 用于请求重新计算 Prototype Author 定义的模板语义输出，并由 adapter 以宿主友好的方式提交这次变化。

这个理解有几个好处：

- 保住了 Prototype 对组件完整交互的定义权；
- 避免 Proto UI 直接滑向“另一个框架”；
- 为不同宿主保留实现空间；
- 也更符合 Proto UI 作为协议层的定位。

因此，后续实践不应围绕“update 是否强大”，而应围绕：

- 哪些动态值属于模板正式消费对象；
- 哪些值变化应当语义上触发重算；
- adapter 至少需要保证哪些一致性。

---

## 当前阶段的边界判断

### 哪些动态呈现更倾向于属于 Prototype 内部定义

以下场景更适合作为 Proto UI 自己的正式职责：

- 当前值文本；
- 选中值摘要；
- 计数/页码/百分比；
- anatomy 中明确存在的 `value` / `summary` / `count` 区域；
- 轻量的条件性辅助文本或局部结构变化。

这些场景如果一律交给宿主扩展，Proto UI 对组件完整交互的控制力会明显下降。

### 哪些动态呈现可以暂时留给宿主扩展

以下场景可以暂时不强求 Proto UI 直接承担：

- 高度定制的复杂视觉呈现；
- 宿主专属布局增强；
- 依赖宿主特化能力的复杂 render hook；
- 不属于组件核心交互定义的展示强化。

换句话说：

> 宿主扩展应是增强路径，而不应成为 value 类组件的默认实现路径。

---

## 关于 `*-value` 抽象与 asHook 的阶段性结论

这一议题目前暂时搁置。

原因不是它不重要，而是：

- 当前仍缺少足够实践来判断它是否存在稳定抽象；
- 现阶段更需要通过多个原型观察重复模式，而不是提前发明统一 API；
- `value` 这个词本身语义负载过重，过早抽象容易把不同层面的能力混在一起。

目前更稳妥的做法是：

- 先把 `*-value` 视为一类值得重点观察的实践对象；
- 记录其在不同组件中的：
  - 值来源；
  - 显示值规约方式；
  - 呈现挂点；
  - 是否需要 update；
- 等重复模式足够清晰后，再决定是否存在值得抽象的内部能力或特权级别 asHook。

当前不宜把“抽象 asHook”作为先决任务。

---

## 每个原型实践时建议记录的问题

为了避免最后只得到“做出来了 / 没做出来”的模糊结论，建议每个样本都按同一套观察项记录。

### 1. 值由谁消费

- 仅供 expose / style system 消费？
- 由模板内部直接消费？
- 同时存在两种消费路径？

### 2. 值变化后谁负责触发更新

- state.set 是否足以驱动所需回显？
- 是否需要显式调用 `run.update`？
- 多次值变化如何合并？

### 3. 模板更新的影响范围

- 仅文本变化？
- 属性变化？
- 局部结构切换？
- 一个值是否驱动多个模板位置联动？

### 4. 宿主耦合程度

- 是否依赖宿主原生输入模型？
- 是否容易被宿主调度差异放大？
- adapter 是否需要额外补偿逻辑？

### 5. 是否暴露出新的稳定概念

- 是否出现重复的 anatomy part？
- 是否出现重复的值来源/值格式化模式？
- 是否出现重复的受控更新需求？

只有在这些模式多次重复后，再考虑进一步抽象。

---

## 当前阶段的实践目标

这一轮工作的成功标准，不是“实现一批看上去常见的表单组件”，而是：

1. 让 `run.update` 从“几乎未被使用的名义能力”变成有明确样本支撑的正式能力；
2. 区分 expose-driven 与 render-consumed 两类 value 场景；
3. 验证 Proto UI 是否能在不滑向宿主级框架的前提下，保有对 value presentation 的定义权；
4. 为后续是否需要：
   - 正式化 `value` / `summary` / `count` 类 anatomy，
   - 或抽象出内部能力 / 特权级别 asHook  
     提供真实样本基础。

---

## 当前阶段的总结

现阶段关于 value 类组件，可以先收束为以下几条判断：

- `run.update` 不能长期停留在理论存在、实践缺席的状态；
- 不是所有带值组件都天然是 update 探针；
- 真正优先探索的，是模板明确消费动态值的场景；
- Stepper 与一类 `*-value` / summary / count 原型，比 Slider / Rating 更适合作为第一批样本；
- 宿主扩展很重要，但不能默认替代 Prototype 对组件完整交互的定义；
- 关于 `*-value` 是否存在稳定抽象、是否值得抽出 asHook，目前结论应保持开放，等待更多实践证据。

当前最稳的路线不是提前抽象，而是先通过一批 carefully chosen 的原型，把边界真正踩出来。
