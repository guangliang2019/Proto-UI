# 2026-04-13 v0 首次公开发布范围与优先级

> Internal record. Not normative. 本文用于收敛 Proto UI 首次公开发布前必须证明什么、不必证明什么，以及剩余工作的优先级划分。

---

## 1）背景

Proto UI 当前已经在多个层面上积累了相当多的实现工作，包括：

- `internal/contracts` 下的协议契约
- React / Vue / Web Components 的 runtime 与 adapter 实现
- `packages/prototypes/base` 与 `packages/prototypes/shadcn` 下的原型库
- `apps/www` 下的文档与演示内容
- `scripts/release` 下的发布脚本

但与此同时，第一次发布前剩余的工作仍然分散在多个方向上：

- quick start 与最小使用闭环
- 原型库补齐
- 文档建设
- 契约编目与契约测试编目
- CLI 与 npm package 发布
- 组织管理 / CI / CD / 项目后勤
- 适配器覆盖扩张
- 网站与社交媒体运营

当前真正缺的，不只是“再做更多”，而是更清晰的发布边界。

如果没有这条边界，第一次发布就很容易从“证明 Proto UI 这套模型是真的”滑向“试图把整个生态一次性做完”。

---

## 2）已观察到的事实

### 2.1 当前实现已经明显超过玩具阶段

截至 2026-04-13：

- 工作区测试通过，结果为 `153 passed | 3 skipped`
- 测试总数为 `558 passed | 34 todo`
- React / Vue / Web Component 三个 adapter 都已经有实际测试覆盖
- base 与 shadcn 两套 prototype library 都已存在且已有测试

因此当前状态并不是“什么都还没准备好”。

更准确地说，当前更像是：

- 核心语义层已经有了可观的实现基础
- 跨 adapter 的验证已经是真实存在的
- 公开发布所需的 package、上手路径和范围收口，反而落后于实现深度

### 2.2 当前 Quick Start 还不是一个真实的对外路径

现有 `quick-start.mdx` 明确写了：

- CLI 还没准备好
- 文中的命令现在不应该真的执行
- 这篇文档描述的是“未来预期流程”，不是“当前可用流程”

这意味着当前最关键的对外承诺其实还没有闭环。

### 2.3 发布脚本和包元数据已经存在，但 publish readiness 还不够干净

仓库中已经有：

- release scan
- staging / publish 脚本
- adapters / prototypes / runtime / modules / core 等 package 的元数据

但目前 `release:scan` 仍然会报出较多发布层面的风险，主要包括：

- 很多 package 仍然直接导出 `src/*.ts`
- 有些 package 仍缺少 package README

这说明 package 发布能力已经在建设中，但还没有到“可以放心对外 ship”的状态。

### 2.4 开源协作骨架已存在，但 CI/CD 仍然偏轻

当前仓库里已经有：

- `CONTRIBUTING.md`
- adapter / prototype / docs 的 issue template

但目前 `.github/workflows` 下看不到明显的 workflow 文件。

这意味着对外协作入口已经有了，但面向公开发布的基础自动化仍然偏弱。

### 2.5 当前 adapter 覆盖范围已经超过首次发布的最小证明负担

对第一次公开发布来说，以下三种 Web adapter 已经是很有说服力的范围：

- React
- Vue
- Web Components

在 v0 阶段继续扩充 adapter，当然能增加表面覆盖，但它对“Proto UI 这套模型是否成立”的证明价值，已经没有“把首个官方使用闭环跑通”那么高了。

---

## 3）问题定义

Proto UI 现在需要一份明确的发布决策，回答三个问题：

1. 第一次公开发布成功，最低要满足什么？
2. 哪些工作是真正的发版阻塞项，哪些只是质量增强项？
3. 哪些看起来很重要、但应该明确推迟到发布之后再做？

如果现在不把这件事定下来，项目很容易掉进两个相反的风险：

- 技术内核已经很强，但首个用户路径其实是断的，却提前发布
- 因为一直追求生态完整度，导致发布无限期拖延

---

## 4）当前决策

### 4.1 第一次公开发布要证明的是“首个官方使用闭环”，不是“生态完整”

v0 第一次公开发布应当被视为成功，只要它能证明以下几点：

- 一个新的外部用户，能沿着一条官方路径真正把 Proto UI 跑起来
- 这条路径能体现 Proto UI 的跨 adapter / 协议层价值
- 这条路径背后有 docs、packages 和 tests 支撑
- 对外口径能如实描述当前边界与非目标

它**不需要**在首次发布时证明：

- 很大的组件广度
- 超过当前三种 adapter 的覆盖范围
- 非常成熟的贡献者自动化
- 很完整的社区运营体系
- 一个功能非常完整的 CLI 生态

### 4.2 优先级应由“是否影响发版”定义，而不是由“长期是否重要”定义

建议本轮发版采用三档优先级：

- `P0`：发版阻塞项；未完成则不应发第一次公开版
- `P1`：强烈建议在发版前完成，但如果核心闭环已经成立，可接受部分缩减
- `P2`：明确后置；有价值，但不需要进入第一次公开发布门槛

### 4.3 各工作方向的优先级建议

| 工作方向 | 优先级 | 原因 |
| --- | --- | --- |
| 最小使用场景闭环（Quick Start 可跑通） | P0 | 这是对外证明路径。没有它，发布叙事就是不成立的。 |
| 文档建设 | P0 | 第一次发布很大程度上是在为已经存在的技术内核建立可理解、可信任的入口。 |
| CLI / 测试 / npm package 发布与管理 | package 发布路径是 P0，CLI 本身是 P1 | package 必须可安装；CLI 不应在无必要时升级为主要阻塞项。 |
| 契约编目、契约测试编目 | P1 | 对后续贡献者扩展和 review 很重要，但不必在发版前追求绝对完整。 |
| 原型库建设 | P1 | 需要冻结一个最小可信集合，而不是继续无限扩张。 |
| 组织管理、CI/CD 等后勤建设 | P1 | 要有基础自动化与协作入口，但不需要一次做到成熟运维体系。 |
| 适配器覆盖范围扩充 | P2 | 当前三种 adapter 已足够证明 v0 的成立性。 |
| 网站运营、社交媒体运营 | P2 | 首发时需要有消息与入口，但不需要完整的长期运营节奏。 |

### 4.4 Quick Start 不应该被“完整 CLI”卡死

这是本记录里最重要的范围判断。

建议规则如下：

- 第一次公开发布必须有一个真实可跑的 Quick Start
- 但这个 Quick Start **不必**依赖一个成熟的官方 CLI

更稳妥的发版策略是：

- 先让 packages 变得可安装
- 先提供一条官方支持的手动接入路径
- CLI 只在它能以“最小可用形态”及时落地时再进入发版路径

这样，发版负担就会被绑定在“用户是否成功”，而不是被绑定在“工具是否足够野心化”上。

如果团队坚持对外故事必须是“Proto UI 从 CLI 开始”，那 CLI 就会立刻升为 `P0`。

但这应该被理解为一个产品叙事选择，而不是一个技术上的必然要求。

### 4.5 第一次发布门槛必须比当前愿望清单更窄、更硬

第一次发布建议只强制要求以下六件事：

1. 一条官方可运行的 onboarding 路径
2. 一个官方推荐的 onboarding adapter
3. 一套小而可信的 prototype 集合
4. 一套与真实安装/使用流程一致的文档路径
5. 一套可重复执行的 package staging / publish 流程
6. 一条会在 PR 上自动验证仓库健康度的基础 CI 路径

除此之外的工作，都应该反过来证明自己为什么需要挤进这六个门槛之前。

## 5）发布门槛定义

### 5.1 Gate A：存在一条真实的官方上手闭环

最低可接受状态：

- Quick Start 不再写“现在还不可安装”
- 用户能从 docs 出发，最终在本地跑起一个组件
- 文档中的命令或代码样例与实际发版路径一致
- 这条路径至少有一个 smoke 级别的验证

建议首发路径：

- 选择一个官方推荐 onboarding 技术栈，优先 React
- 选择一个 styled prototype 和一个 headless prototype 作为示范
- 保持路径刻意简短，不追求花哨

### 5.2 Gate B：prototype 集合已经冻结到“足够可信”

最低可接受状态：

- 当前用于文档与 demo 的 base / shadcn prototypes 已稳定
- 每个被列为“首发支持”的 prototype 都有测试
- 文档能明确列出当前支持的 prototype 集合
- 未支持或未完成的区域被明确标记

第一次发布不要求有一个很大的目录。

它要求的是一个可信的目录。

### 5.3 Gate C：文档真实、一致、够用

最低可接受状态：

- README 与 docs site 说的是同一个故事
- 当前 status / roadmap 与 Quick Start 彼此一致
- 有清晰的普通用户入口
- 有清晰的贡献者入口
- 有清晰的非目标与风险边界说明

### 5.4 Gate D：package 发布在操作层面已经真实可行

最低可接受状态：

- 对外 package 集合已被显式选定
- 针对这组 package 的 release staging 可成功执行
- 目标发布 package 上的 README / export / metadata 阻塞项已清理
- 发布步骤有文档，并且可重复执行

这个门槛关注的是“可操作的发布信心”，而不是“第一次就要把所有内部 package 都打磨成独立 public surface”。

### 5.5 Gate E：基础项目后勤已具备

最低可接受状态：

- PR 时至少自动跑 type check 与 tests
- issue intake 仍然清晰可用
- 发布职责和基础 checklist 已写下来

第一次发布不需要复杂的 dashboard、治理制度或全套自动化。

---

## 6）明确允许后置到发布之后的内容

以下内容除非意外变得非常便宜，否则都可以安全地放到发布之后：

- React / Vue / Web Components 之外的新 adapter 家族
- 超出首发集合的 prototype 大规模扩张
- 更完整、更复杂的 CLI 命令面
- 契约与测试目录的深度打磨
- 更高级的 CI/CD 编排，如多阶段 release promotion
- 高频社交媒体节奏与持续性内容运营
- 超出首发消息所需范围的品牌与传播工作

---

## 7）建议推进顺序

### Phase 1：先把第一条官方路径做实

- 决定首发 onboarding 路径
- 决定首发 package 集合
- 清理这批 package 的 publish blocker
- 重写 Quick Start，让它与现实一致

### Phase 2：再把首发表面做可信

- 冻结首发 prototype 集合
- 增补契约 / 契约测试目录索引
- 补基础 CI
- 审核 README、status、roadmap、homepage 的口径一致性

### Phase 3：做发布支持，而不是继续扩范围

- 准备网站首发页面与入口
- 准备一版 announcement thread / post
- 准备贡献者 intake 说明

任何新工作流如果不能直接增强“官方上手闭环”，都不应该插队到 Phase 1 前面。

---

## 8）结果与影响

这份决策有意缩窄了“什么叫准备好发布”。

它带来的影响是：

- 一些看起来很诱人的生态扩张工作，会被明确延后
- 文档真实性比文档广度更重要
- package / install 的现实可用性高于工具雄心
- adapter 扩张不再允许与 onboarding 闭环抢优先级

这对 v0 是正确的。

Proto UI 的第一次发布，不需要看起来“已经完整”。

它需要的是：真实、可跑、可信。
