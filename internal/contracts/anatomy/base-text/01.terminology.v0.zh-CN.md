# Anatomy 术语底本（v0）

本文用于定义 Anatomy v0 的核心术语、职责边界与非目标。

Anatomy 在 Proto UI 中表示一套**结构语义系统**，用于描述某个 prototype family 的组成部分、结构关系、基础不变量，以及同一结构域内的受限观察能力。

Anatomy **不是**装配器、也不是组合器。它不负责创建 part、不负责改写真实树结构，也不负责补全默认 prototype。

---

## 1. 基本定位

Anatomy 提供的能力包括：

- 描述一组具名结构角色及其关系
- 对实际结构进行匹配、校验与诊断
- 在同一 anatomy 域内提供受限的 runtime 查询能力
- 检查某个角色是否满足额外要求
  - 例如某个 `as-hook`
  - 某个 exposed method / state / event

Anatomy 不提供的能力包括：

- 自动创建缺失 part
- 自动搬运、重排、包裹现有 part
- 修改 logical tree / host tree
- 替代 `context` / `event` / `props` 成为新的通用通信系统

---

## 2. 核心术语

### 2.1 Anatomy Family

**Anatomy Family** 是一组可被多个 prototype 共享的解剖学语义坐标系。

Family 定义：

- 有哪些 `role`
- 这些 `role` 之间允许或期望存在哪些 `relation`
- 哪些约束属于基础 `invariant`

Family 应尽量：

- 稳定
- 宽松
- 可拓展
- 面向语义，而不是面向某一种具体 UI 实现

示例：

- `Select Family`
- `ScrollArea Family`

### 2.2 Role

**Role** 是 family 内一个具名的组成部分语义位。

Role 表示“一个 part 在 anatomy 中承担什么职责”，而不是“这个具体 prototype 是谁”。

示例：

- `root`
- `trigger`
- `content`
- `item`
- `scrollbar`

### 2.3 Part

**Part** 是一个实际加入某个 anatomy family 的 prototype instance。

`part` 是 runtime 中的真实成员；`role` 是该成员在 anatomy 中的语义身份。

可以说：

- “这个实例是 `item` role 的一个 part”

### 2.4 Relation

**Relation** 是 role 与 role 之间的结构关系约束。

Relation 用于表达：

- 父子
- 归属
- 配对
- 依附
- 数量关系

Relation 只描述期望结构，不负责修改现有结构。

### 2.5 Invariant

**Invariant** 是 family 级别的不变量。

违反 invariant 表示实际结构已经不能再被视为该 family 的合法成员，或至少已经失去该 family 的核心语义。

Invariant 应优先表达：

- 语义本质
- 最低成立条件

而不应优先表达：

- 某一套默认视觉实现
- 某一类 authoring 偏好

### 2.6 Profile

**Profile** 是建立在 family 之上的具名收紧约束集。

Profile 用于表达：

- 某个具体 prototype 默认期望的 anatomy
- 某种常见交互变体的结构偏好
- 比 family 更严格的计数、requirement 或诊断规则

Profile 不应推翻 family 的基础语义。

可以理解为：

- family 负责提供稳定、宽松的结构坐标
- profile 负责为具体 prototype 或具体用法收紧规则

示例：

- `select/default`
- `select/range`

### 2.7 Requirement

**Requirement** 是某个 role 需要满足的额外能力要求。

Requirement 的检查对象可以包括：

- 某个 `as-hook`
- 某个 expose key
- 某类 expose 能力
- 某个结构标记

Requirement 属于结构检查的一部分，而不是行为注入机制。

### 2.8 Anatomy Domain

**Anatomy Domain** 是某个 family 在一棵 logical tree 中的具体作用域实例。

可以把它理解为：

- “这一组互相关联的 parts，构成了一个具体的 anatomy 域”

例如一个实际存在的 `Select` 及其相关 parts，就构成一个 `Select Family` 的 domain。

### 2.9 Presence

**Presence** 表示某个 role 在当前 anatomy domain 中是否实际存在。

Presence 是 runtime 查询与诊断的基础观察结果之一。

### 2.10 Anatomy View

**Anatomy View** 是 `run.anatomy` 暴露出的受限观察视图。

它只应用于：

- 查询某个 role 是否存在
- 查询某个 role 是否满足 requirement
- 读取某个 role 已显式 expose 的能力或状态

Anatomy View 不是：

- 实例句柄
- host 节点引用
- 内部状态访问入口
- 通用跨组件通信系统

---

## 3. 推荐的层级理解

建议按以下层级理解 Anatomy：

1. `family`
   - 提供稳定的结构语义坐标
2. `role`
   - 提供结构中的职责名
3. `profile`
   - 为某个具体 prototype 或具体用法收紧结构期望
4. `part`
   - 在 runtime 中实际加入某个 domain 的成员
5. `view`
   - 对 domain 中其他 role 的受限观察方式

---

## 4. 术语边界与职责分离

### 4.1 Anatomy 与 Template

Anatomy 不定义 render output，也不改变 Template 的合法性边界。

即使某个 family 定义了丰富的 role 与 relation，Proto UI 仍然**不因此允许**在 Template 中直接进行 prototype-level composition。

### 4.2 Anatomy 与 Context

Context 是通用的数据通路；Anatomy 不是。

Anatomy 只允许基于已声明的结构角色做受限观察，不应承担通用 provider / consumer 通讯职责。

### 4.3 Anatomy 与 Expose

Expose 是 part 对外显式公开能力的通路；Anatomy 可以在受限范围内读取这些 expose。

Anatomy 不新增一套独立的输出系统，而是复用 expose 作为可观察面的主要来源。

### 4.4 Anatomy 与 Assembly

Anatomy 不负责装配。

凡是涉及以下行为，均不属于 Anatomy：

- 创建新的 prototype instance
- 注入默认 part
- 搬运已有 part 的真实位置
- 重写或物化缺失结构

---

## 5. 诊断语义的建议分层

Anatomy 的诊断建议至少区分两层：

### 5.1 Invariant Diagnostic

违反 family 的基础 invariant。

这类问题说明该结构已经偏离 family 的核心语义，通常应视为更高等级的问题。

### 5.2 Profile Diagnostic

违反某个 profile 的收紧规则。

这类问题说明当前结构不符合该 prototype 或该用法的推荐 anatomy，但不必然意味着 family 语义已经失效。

---

## 6. 非目标

Anatomy v0 当前不追求：

- 自动补全默认结构
- 自动 materialize 默认 part
- 通用 override / merge 任意 family 定义
- 运行时动态改写 anatomy 语义
- 成为新的 instance registry

---

## 7. 一句话定义

`Anatomy Family` 定义一组 `Role` 及其 `Relation` 和 `Invariant`；`Profile` 在 family 之上为具体 prototype 收紧约束；runtime 通过 `Anatomy View` 对当前 `Domain` 中的 `Part` 做受限观察。
